# 유현모바일 목업 구현 문서

## 목적

이 프로젝트의 현재 목표는 **백엔드 API 연결 없이** 모바일 화면 시연이 가능한 Flutter 목업 앱을 만드는 것입니다.  
즉, "기능 검증"이 아니라 "화면 흐름/디자인 검토"에 초점을 둔 구현입니다.

---

## 구현 방식 요약

- 프레임워크: Flutter
- 상태관리: `flutter_riverpod`
- 라우팅: `go_router`
- 네트워크 계층: `dio` (구조만 준비, 데모 모드에서는 미사용)
- 인증 저장소: `flutter_secure_storage` (구조만 준비, 데모 모드에서는 미사용)

핵심은 다음 2가지입니다.

1. **백엔드 0호출 데모 모드**
2. **목업 데이터 기반 화면 렌더링**

---

## 백엔드 0호출 (Demo Mode)

현재 앱은 시연용으로 백엔드 호출을 하지 않습니다.

- 로그인 시 `/api/auth/login` 호출 없음
- 대시보드/센서/알람 등 모든 화면에서 HTTP 요청 없음
- 데이터는 앱 내부 목업 데이터(`lib/mock/mock_data.dart`)를 사용

인증은 `auth_controller.dart`에서 상태값만 변경합니다.

- 로그인 버튼 클릭 -> `isLoggedIn = true`
- 로그아웃 버튼 클릭 -> `isLoggedIn = false`

라우팅 가드는 `app.dart`에서 처리합니다.

- 로그인 전: `/login`만 접근 가능
- 로그인 후: `/dashboard` 이동
- `/qr/:id`는 공개 접근 허용

---

## 목업 데이터 설계 방식

초기에는 문자열 목록만 사용했지만, 시연 품질을 위해 **화면별 모델 구조화**를 적용했습니다.

파일: `lib/mock/mock_data.dart`

- `DashboardMetric`
- `SensorSummary`
- `AlarmItem`
- `SiteItem`
- `FileItem`
- `UserItem`

예시:

- 알람: 센서코드, 센서명, 심각도, 메시지, 상대시간
- 현장: 현장명, 위치, 담당자, 센서수, 주의수
- 파일: 파일명, 용량, 업로더, 업로드 시간
- 사용자: 이메일, 권한, 연락처, 활성 상태

장점:

- 화면 컴포넌트가 문자열 파싱 없이 바로 사용 가능
- 나중에 실제 API 응답 모델로 전환하기 쉬움

## 디자인 반영 전략

웹 프론트와 최대한 유사한 톤을 맞추기 위해 디자인 토큰/공통 컴포넌트를 만들었습니다.

### 1) 공통 색상 토큰

파일: `lib/core/theme/app_colors.dart`

- surface/page/card
- line/lineStrong
- text(ink/sub/muted)
- brand
- 정상/주의/위험/오프라인 상태 컬러

### 2) 공통 테마

파일: `lib/core/theme/app_theme.dart`

- 카드/입력창/앱바 기본 스타일 통일
- 프로젝트 전체 스타일 일관성 확보

### 3) 공통 UI 위젯

파일: `lib/features/common/presentation/geo_widgets.dart`

- `GeoCard`: 웹의 `geo-card` 톤 카드
- `SectionTitle`: 좌측 포인트 바가 있는 섹션 타이틀
- `StatusBadge`: 정상/주의/위험/오프라인 뱃지

## 페이지별 목업 구현 포인트

### 로그인 (`login_page.dart`)
- GM 로고 박스
- 카드형 로그인 폼
- 에러 안내 박스
- 테스트 계정 안내

### 대시보드 (`dashboard_page.dart`)
- KPI 카드 그리드 (상단 컬러바, 큰 숫자)
- KPI 클릭 필터 인터랙션
- 최근 알람 카드 리스트

### 센서 목록 (`sensors_page.dart`)
- 상태 필터 칩 UI
- 센서 카드 + 상태 뱃지 + 마지막 수신 시간

### 센서 상세 (`sensor_detail_page.dart`)
- 탭 구조: 정보 / 트렌드 / 로그
- 정보 요약 카드
- 기간 버튼 + 차트 목업 영역
- 시간별 로그 리스트

### 알람/현장/파일/사용자
- 같은 카드 톤으로 통일
- 뱃지/상태 강조 UI 적용

## 향후 실제 API 전환 시 권장 방식

현재 구조는 API 전환을 염두에 두고 분리되어 있습니다.

권장 순서:

1. `mock_data.dart`를 대체할 Repository 인터페이스 정의
2. `demo repository`와 `api repository`를 분리 구현
3. Riverpod Provider에서 주입 대상을 스위칭
4. 화면 코드는 최대한 수정 없이 데이터 소스만 교체

즉, 현재 목업 코드는 "버리는 코드"가 아니라 **전환 가능한 UI 자산**을 목표로 구성되어 있습니다.
